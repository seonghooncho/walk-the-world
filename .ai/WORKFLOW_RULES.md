# 개발 워크플로우 규칙

## 근본 목적
모든 산출물(이슈, PR, 작업 문서, 트러블슈팅 기록)은 지엽적 제약보다 근본 목적을 먼저 쓴다.

- 첫 기준은 "왜 이 작업을 하는가"다.
- 이 저장소의 기본 목적은 운영 관측 복구와 원인 추적 속도 향상이다.
- 세부 설정, 자잘한 이슈, 구현 디테일은 목적 달성에 직접 연결될 때만 기록한다.

## 비목적
아래 항목은 독립 목적처럼 문서 전면에 오지 않도록 제한한다.

- "설정값 변경 자체", "자잘한 오류 수정 자체"를 주 목표로 작성하지 않는다.
- 목적과 연결되지 않은 구현 디테일을 과도하게 서술하지 않는다.
- 증상 나열만 하고 왜 이 작업을 하는지 빠뜨리는 기록을 남기지 않는다.

## 1) 순서 강제
아래 순서를 반드시 지킨다.

1. 요청 수신 직후 아래 4개만 출력한다.
   - (1) Request Summary
   - (2) Plan
   - (3) Risks
   - (4) Approval Needed
2. 사용자 승인 전에는 실제 실행을 금지한다.
   - 파일 생성/수정
   - 커밋
   - PR 생성
   - 외부 시스템 변경
3. 승인되면 (5) Checklist를 작성한다.
4. Checklist 확정 후 GitHub Issue를 생성한다.
5. Issue 번호로 브랜치를 생성한다.
   - 규칙: `feature/<slug>-<issueNumber>`
6. 작업을 수행한다.
7. 변경 요약과 검증 결과를 먼저 보고하고 커밋 승인을 받는다.
8. 승인되면 커밋한다.
9. PR을 생성한다.

## 2) 출력 포맷
승인 전 출력은 아래 4개 섹션만 사용한다.

### (1) Request Summary
- 요청 요약 1~3줄
- 근본 목적 1줄
- 비목적 1줄

### (2) Plan
- 실행 계획 2~5개
- 각 계획은 목적과의 연결을 포함한다.

### (3) Risks
- 예상 리스크와 대응

### (4) Approval Needed
- `yes/no`로 답할 수 있는 승인 질문 1개

승인 후 출력은 아래 섹션을 사용한다.

### (5) Checklist
- 확정된 TODO 목록

### (6) Execution Log
- 핵심 진행 로그만 간단히 기록

### (7) Verification
- 실행한 검증 명령과 결과 요약

### (8) PR Notes
- PR 본문에 들어갈 핵심 요약
- 요약은 목적 달성 관점으로 작성한다.

## 3) 위험 작업 승인 규칙
아래 작업은 명시적 승인 없이는 절대 실행하지 않는다.

- 배포
- DB 스키마 변경
- 마이그레이션
- 데이터 변경/삭제
- 복구가 어려운 운영성 작업

## 4) 커밋/PR 작성 규칙
- 제목/본문은 짧고 사실 중심으로 작성한다.
- 군더더기 문구를 넣지 않는다.
- 이모지를 사용하지 않는다.
- 과도한 서술을 하지 않는다.
- 변경 수단보다 변경 목적을 먼저 쓴다.
- 세부 제약은 "왜 필요한지"가 설명되지 않으면 본문에 쓰지 않는다.

## 5) 문서 작성 강제 규칙
아래 규칙을 이슈, PR, Markdown 문서 전반에 적용한다.

- 필수 섹션: `## 근본 목적`, `## 비목적`
- 필수 내용: 각 섹션은 `-` 같은 placeholder가 아닌 실제 문장으로 작성한다.
- 트러블슈팅 문서: 증상/설정보다 "어떤 관측 공백을 복구하는지, 원인 추적 시간을 어떻게 줄이는지"를 먼저 기록한다.
- 자동 검증: `.ai/scripts/verify-purpose-focus.sh`와 `.github/workflows/purpose-guard.yml`로 검사한다.
